## Detailed description — EN

```
TextHumanize brings a full offline text engine into your browser — and puts it right where you write. Select text or focus an editor and instantly:

✨ HUMANIZE — reduce AI-typical style signals: formulaic connectors, bureaucratic vocabulary, uniform sentence rhythm. Replace the text right on the page (input fields, text areas and rich text editors) or copy it.

🔍 CHECK — get an honest AI-style score (0–100%) from a 15-metric statistical detector: burstiness, entropy, AI-pattern density, passive voice, rhythm, perplexity and more.

🎭 TONE · 📖 READABILITY · 🔀 PARAPHRASE · 🧬 STYLE DNA — analyze and shift formality, measure standard readability scores, reword text, and read its stylometric fingerprint.

🧹 CLEAN — strip hidden watermarks: zero-width characters, Unicode steganography, homoglyphs, trailing-space patterns, plus a Kirchenbauer-style statistical test.

🌐 AI-BUILT SITE DETECTION — open the popup on any site and a living score orb tells you whether the whole page was likely made with AI. It reads only the page's own structural code — the generator meta tag, asset paths, component attributes and class names, fonts and the palette — to recognise 80+ website builders and CMS platforms, including the newer generation of AI site generators and the component stacks that AI coding assistants emit. It also counts AI-watermarked images. The evidence is listed openly, and 👍/👎 corrects the verdict per site.

🖼️ IMAGE AI CHECK — hover or right-click any image to check it for AI-generation markers (C2PA provenance, XMP and EXIF metadata, and generator signatures left by common image models). Scanned 100% on your device. Honest verdicts — "AI-generated", "Authentic (C2PA)", or "No markers (not a guarantee)".

WHERE YOU WORK

WHY TEXTHUMANIZE

⚡ Fast. 20–300 ms per paragraph. Pure algorithms, no model downloads.
🌍 25 text languages with automatic detection. Interface in 14 languages.
🎛 Intensity 0–100, 9 style profiles, reproducible output.
🆓 Free and open source (MIT).

Shortcuts: Ctrl/Cmd+Shift+H — humanize selection · Ctrl/Cmd+Shift+U — check selection.
```

## Single purpose (for review)

```
Local, offline processing of the page the user is on: naturalizing/scoring user-selected text, removing hidden watermarks, checking images for AI-generation metadata, and fingerprinting the page's own code to tell if the site was built with AI. All processing happens on-device.
```

## Permission justifications

- **contextMenus** — right-click "Humanize / Check / Tone / Clean" actions and the image check.
- **storage** — stores user settings, the per-site block-scan allow-list, and local usage/feedback counts. No text or page content is stored.
- **activeTab** — lets the toolbar/context-menu action interact with the tab the user invoked it on.
- **scripting** — re-injects the panel content scripts into tabs open before installation, only when the user invokes the extension there.
- **Content script on all sites** — the selection bubble, in-editor chip, image badge and the on-load site-code fingerprint must be available where the user works. They read a selection or image only when the user explicitly triggers an action; the site fingerprint reads only structural page code (meta/generator, asset hosts, class/attribute names, fonts) — never page text or personal content.
- **host permission `<all_urls>`** — lets the service worker fetch image bytes to inspect AI-generation metadata on-device, and is covered by the same "all sites" access the content scripts already declare. No page data is sent anywhere.
- **optional `clipboardWrite`** — copying results to the clipboard.

## Privacy

- Privacy policy URL: `https://github.com/ksanyok/texthumanize-extension/blob/main/PRIVACY.md`
- Data collection: **anonymous usage events only** (which tools are used; never text, URLs or page content). On by default, disclosed, one-click off. Text and image processing are fully on-device.

### Regenerating the screenshots

`scripts/shots.html` is the capture stage: 1280×800, marketing copy on the left,
a live iframe of the real popup/options on the right with mock state injected
same-origin (`paint()`). Edit the `SHOTS` map there to change copy or state.

```sh
cd texthumanize-extension
python3 -m http.server 8642 &                 # same-origin iframe access
CHROME="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
for n in 1 2 3 4; do
  ("$CHROME" --headless --disable-gpu --no-sandbox --hide-scrollbars \
     --force-device-scale-factor=1 --user-data-dir="/tmp/thx-shots/p$n" \
     --window-size=1280,800 --virtual-time-budget=4000 \
     --screenshot="/tmp/thx-shots/shot-$n.png" \
     "http://127.0.0.1:8642/scripts/shots.html?n=$n" >/dev/null 2>&1 &)
  for i in $(seq 1 25); do [ -s "/tmp/thx-shots/shot-$n.png" ] && break; sleep 1; done
  pkill -f "user-data-dir=/tmp/thx-shots/p$n"
  sips -s format jpeg -s formatOptions 92 "/tmp/thx-shots/shot-$n.png" --out "store/v3/shot-$n.jpg"
done
```

⚠ Use classic `--headless`, not `--headless=new` (it hangs here), and note macOS
has no `timeout` command — hence the background-launch + poll pattern.
Screenshots must carry no third-party product names (keyword-stuffing rule above).

The promo tiles come from `scripts/promo.html` (standalone, no iframe) with the
same capture loop at their own sizes:

```sh
#   ?n=small   → --window-size=440,280
#   ?n=marquee → --window-size=1400,560
```

Its orb shadows are fractions of `--d` rather than the fixed px values in
`popup/popup.css`, so the sphere keeps its volume at both 96px and 300px.

## Rejection history — read before resubmitting

| Version | Violation | Cause | Fix |
|---------|-----------|-------|-----|
| 3.1.x (earlier) | Keyword stuffing (*Yellow Argon*) | Third-party product names listed in the description | Every brand list removed; see the warning at the top of this file |
| 3.1.9 | Content policy — functionality not reproducible (*Red Potassium*) | The popup's main **Humanize** button threw `i18n.getMessage(...): No matching signature` | Fixed in 3.1.10 (below) |

**What actually broke in 3.1.9.** `renderResult()` built its heading with
`t(mod?.i18n)`. The main Humanize button calls `runOp('humanize', …)` without a
module, so that optional chain passed `undefined` into `chrome.i18n.getMessage`,
which rejects a non-string key. The exception was swallowed by `runOp`'s catch
and surfaced as an error banner where the humanized text should have been — so
the single headline feature failed on the first click.

It survived testing because the `?page=1` web preview and the unit tests both run
with `IS_EXTENSION === false`, where `t()` takes a plain-object fallback path that
happily accepts `undefined`. **The bug only ever existed inside a real extension.**

Guards added in 3.1.10:
- `t()` in both `popup/bridge.js` and `content/shared.js` now rejects a non-string
  key and coerces substitutions to strings, so a bad key degrades to a missing
  label instead of taking a feature down.
- `runOp()` resolves the module from the op, so the heading is correct too.
- `scripts/e2e-smoke.mjs` drives the **real loaded extension** and exercises every
  advertised module. `npm run package` runs it automatically (`prepackage`), so a
  build that fails in-extension can no longer be packaged.

## Detection quality (v3.3.0)

Driven by a 2026 evidence review (saved in `research/`) and measured with
`npm run bench:detector` over a labelled corpus (public-domain human prose vs AI):

- **Site detection** now reads the page's own default subdomain — the cheapest,
  most certain signal (a site on `*.lovable.app` / `v0-*.vercel.app` /
  `*.framer.website` is near-certainly that platform). Plus free-tier badges,
  live-verified 2026 code markers, new builders (Base44, Emergent), and a
  separate weak "vibe-host" tier for generic deploy hosts.
- **Image detection** fixed a false-positive class: a human photo carrying a
  `Description`/`Title`/`Comment` metadata chunk was reported as AI. Generic
  keys now require an actual generation-parameter value; ComfyUI/InvokeAI/SwarmUI
  and 2026 generators added.
- **Text detection** added a paraphrase-robust `structure` metric (enumeration /
  list scaffolds), fixed two inverted signals (passive voice, punctuation —
  they were penalising well-edited human prose), added anti-homoglyph
  normalization, and rebalanced weights off dead metrics. The 2025–2026 chat
  "assistant register" is now caught (was scored ~40% human).

Result on the corpus: 0 false positives, the previously-missed assistant register
now reads AI, and false-positive-prone human samples scored lower. Honest ceiling
and no binary "100% AI" verdicts (the product shows a score + evidence list).

## Humanization quality (v3.2.0)

The rewriter used to leave the most recognisable AI tells untouched, because the
detector knew phrases like *"In today's rapidly evolving digital landscape"* and
*«в современном мире»* but no dictionary could remove them. `npm run bench`
measures this over a fixed corpus in 5 languages:

| | before this work | after |
|---|---|---|
| Mean AI score | 64 → 51 | 64 → **49** |
| AI markers left in the output | 20 / 116 | **2 / 116** |
| Mechanical faults (double spaces, broken grammar) | several | **0** |

What changed: a new AI-cliché dictionary in the library (`lang/_ai_cliches.py`,
~250 phrases across en/ru/uk/de/es/pl, exported into `data/langs/*.json`);
multi-word phrases now replace at near-certainty instead of on a coin flip;
burstiness runs before connector replacement so the sentence-merge path works;
and several grammar guards (bare prepositions are never varied, `help` never
precedes a gerund, deletions re-capitalise and collapse whitespace).

`npm run bench` fails the build on any mechanical fault, and it runs as part of
`npm run package`.
